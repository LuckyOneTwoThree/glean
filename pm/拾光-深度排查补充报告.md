# 拾光 / Glean — 深度排查补充报告

> 日期：2026-06-07 | 阶段：深度功能排查与边缘用例检查

在对 `Glean` 核心业务逻辑（P0 需求对齐、重复 API 调用修复等）进行排查并修复后，我们进一步对周边代码和状态管理逻辑进行了二次深度扫雷。主要关注了在特殊条件（如 Web 环境 mock、特定 UI 场景下）的隐性问题。

## 发现与排查结论

### 1. Web端 Mock 数据交互状态未全覆盖 (UI Bug)
**问题描述**：在 `app_state_provider.dart` 中，为了支持 Web 端免 `sqflite` 运行，编写了 `_getMockArticles()` 提供静态数据。虽然实现了 `favoritedArticleIdsProvider` 来支持在 Web 端的点击收藏（即重新生成带状态的 Mock 数据），但是 `is_read`（已读状态）却遗漏了对应的机制。调用 `markArticleRead` 后仅仅触发了重新加载静态的 Mock 数据，UI 中的未读红点或加粗样式无法消失。
**处理状态**：已修复。引入 `readArticleIdsProvider` 在内存中同步 Web 端的已读列表，模拟完整的状态变迁。

### 2. 成本监控功能的上限写死 (逻辑 Bug)
**问题描述**：根据 PRD 5.7 节 “LLM 成本管理” 要求，应用应提供可视化进度条并预设 10 万 tokens 的软上限。但在 `llm_config_screen.dart` 的 `_buildCostMonitor()` 组件中，`budget` 被硬编码 `final budget = 100000;`，而非从当前数据库的 `llm_config`（`LLMConfig.budgetTokens`）中动态读取。
**处理状态**：已修复。修改 `_buildCostMonitor` 读取真实用户配置并进行进度条的响应计算。

### 3. API 测试时的无秘钥拦截不当 (边缘用例)
**问题描述**：`llm_config_screen.dart` 在测试连通性 `_testConnection()` 时，前置无脑拦截了 `_apiKeyController.text.isEmpty` 的情况。考虑到未来如果接入部分免费且无需 Key 的自定义模型（或者完全本地的局域网私有部署模型），强制校验 API Key 会导致测试请求被直接阻断。
**处理状态**：已修复。当 provider 为 "custom" 等自定义情况时，允许 API Key 为空直接下发测试。

### 4. 数据表索引优化建议 (性能预留)
**问题描述**：`database_service.dart` 的 v4 版本虽然针对多张表增加了索引，但 `execution_logs` 表只有 `started_at` 和 `task_type` 索引。由于后期可能高频排查失败日志（`status = 'error'`），当日志量大幅增长后全表扫描可能会影响速度。
**处理状态**：建议保留。暂不需要立即变动代码，后续有相关业务大批量增加时一并做索引升级。
