# 拾光 (Glean) 项目深度分析报告

## 1. 项目概述与技术栈

**拾光 (Glean)** 是一款 AI 驱动的个性化资讯简报应用。该应用主要通过抓取 RSS 源或其他资讯流，结合本地和云端 LLM (大语言模型) 对文章进行评分、筛选和摘要生成，最终为用户呈现一份定制化的每日简报。

**核心技术栈：**
- **框架**: Flutter (SDK: >=3.0.0 <4.0.0)
- **状态管理**: Flutter Riverpod (`flutter_riverpod`, `riverpod_annotation`)
- **本地存储**: `sqflite` (SQLite), `flutter_secure_storage` (安全存储)
- **网络请求**: `dio` (网络层), `xml` (RSS 解析)
- **后台与定时任务**: `workmanager`, `flutter_local_notifications`
- **其他关键库**: `intl` (国际化), `google_fonts` & `flutter_svg` (UI), `share_plus` (分享)

## 2. 项目目录结构分析 (`lib/`)

整个项目的职责划分清晰，遵循了典型的分层架构设计：

*   **`models/`**: 数据模型层。
    *   包含 `article.dart` (文章), `briefing.dart` (简报), `feed.dart` (数据源), `user_config.dart` (用户配置), `llm_config.dart` (模型配置), `score.dart` (评分模型) 等实体类。
*   **`providers/`**: 状态管理层。
    *   核心文件 `app_state_provider.dart` (500多行)。其中集中定义了 Services Providers (依赖注入)、应用状态 (如当前页面、筛选条件、是否完成引导等)，以及异步获取的各类数据 (文章列表、简报列表、配置数据)。对于 Web 端还提供了一套 Mock 数据以作容错和演示。
*   **`screens/`**: UI 视图层。
    *   包括 `home_screen.dart` (首页), `briefing_screen.dart` (简报页), `article_detail_screen.dart` (文章详情页), `onboarding_screen.dart` (新手引导页), `settings_screen.dart` (设置页) 等。
*   **`services/`**: 核心业务逻辑与服务层。
    *   `database_service.dart`: 封装底层 SQLite 操作。
    *   `fetch_service.dart`: 负责从数据源 (RSS等) 抓取原始资讯内容。
    *   `llm_service.dart`: 封装与大模型的 API 交互。
    *   `score_service.dart`: 资讯质量评估服务，可能会调用本地策略或远端大模型打分。
    *   `summary_service.dart`: 内容摘要服务，利用大模型生成一句话概括或核心要点。
    *   `briefing_service.dart`: 简报生成核心服务，串联 Fetch -> Score -> Filter -> Summary 整个工作流。
    *   `schedule_service.dart`: 负责后台周期性任务调度的注册。
*   **`widgets/`**: 通用 UI 组件。
    *   例如 `article_card.dart` (文章卡片), `score_badge.dart` (评分徽章) 等高复用组件。

## 3. 核心业务流程

1. **信息抓取 (Fetch)**: 定时或手动通过 `FetchService` 拉取订阅的 RSS / 数据源。
2. **评估筛选 (Score)**: 利用 `ScoreService` 对抓取到的增量文章进行信息密度和可信度的评分。
3. **内容提炼 (Summary)**: 借助 `SummaryService` 和大语言模型，提取出高分文章的关键摘要 (`summary_one`, `summary_points`)。
4. **生成简报 (Briefing)**: 将所有处理好的数据汇总为每日 `Briefing` 模型并持久化到本地数据库，供首页和简报历史模块展示。

## 4. 当前架构的优点

1. **结构清晰，职责单一**：Services 层处理业务逻辑并与数据库和网络交互，Providers 层做状态管理，Screens 仅负责渲染。
2. **状态管理统一**：使用 Riverpod 对全量数据、UI 状态实现了集中式管理，并巧妙地通过 `_refreshCounterProvider` 的自增来粗放但有效地实现部分数据的强制刷新。
3. **离线与数据沉淀机制**：使用 SQLite 做持久层，配合 Workmanager 进行后台抓取，使用户在弱网环境也能阅读处理好的简报。
4. **适配了多端容错**：在 `app_state_provider.dart` 中，针对 Web 端 (由于缺乏 SQLite/Workmanager 支持) 提供了 Mock 数据兜底逻辑，保证 Web 上的基础运行。

## 5. 后续开发与优化方向建议

针对项目现状，若进行后续开发与优化，可从以下几个维度切入：

### 5.1 架构与性能优化
*   **状态管理细化**：目前依赖一个全局的 `_refreshCounterProvider` 触发大面积重构（Rebuild/Refetch），这在数据量大时会造成 SQLite 反复查询和不必要的性能损耗。建议拆分细粒度的 Provider 依赖，或者采用更现代的 Riverpod Cache 机制及基于 ID 的 StateNotifier。
*   **大文件拆分**：`app_state_provider.dart` 承担了过多的 Provider 声明，随着业务扩充，建议按照功能模块（如：`feed_providers.dart`, `article_providers.dart`, `briefing_providers.dart`）对其进行拆解。

### 5.2 大模型集成深度与体验优化
*   **模型切换与 Fallback**：可以进一步完善 `llm_service.dart` 的配置，实现多模型提供商 (OpenAI, Claude, 智谱等) 的动态切换，并在 API 限流或报错时具备自动降级/重试机制。
*   **流式输出 (Streaming)**：如果当前提炼/摘要环节是阻塞等待返回结果，可以考虑针对长文本和复杂摘要场景引入 SSE (Server-Sent Events) 支持流式输出，提升用户感知性能。
*   **成本管控界面加强**：应用内部虽然有 `llm_cost.dart` 和 `monthlyCostStatsProvider`，可进一步强化前端图表展示，避免用户跑出过高的 API 费用。

### 5.3 业务功能扩展
*   **个性化偏好学习**：记录用户阅读、收藏或忽略的文章数据，反馈给大模型优化后续的评分逻辑 (Prompt 动态调整)，使得简报越来越符合用户的个人口味。
*   **多端同步功能**：目前重度依赖本地 SQLite 存储，后续可以考虑增加通过 CloudKit/iCloud 或自定义服务端同步数据源与收藏记录的功能。
*   **RSS 解析增强**：目前基于 `xml` 库解析，可以引入更加健壮的解析策略（应对非标准 RSS 格式），或增加网页去广告正文提取 (Readability) 能力，解决部分 RSS 只提供摘要的问题。
