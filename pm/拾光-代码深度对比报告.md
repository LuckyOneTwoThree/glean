## 2. 遗漏与缺陷 (Bugs & Issues)

在对比 PRD 和代码实现的过程中，发现了几个较严重的业务逻辑问题，尤其是涉及到 LLM 计费成本的 Bug。

### 🚨 [Bug-01] LLM API 被重复调用两遍，导致 Token 成本翻倍 (严重)
* **表现与原因**：
  在 `llm_service.dart` 中，为了节省请求次数，作者设计了 `scoreAndSummarize` 方法（将评分和摘要请求合并在一个 Prompt 中执行）。
  然而，在 `briefing_service.dart` 的简报生成流程中：
  ```dart
  // 1. 调用评分服务
  await _scoreService.scoreArticle(article.id, config.aiMode); 
  // 2. 调用摘要服务
  await _summaryService.summarize(article.id, config.aiMode);
  ```
  `score_service.dart` 内部会调用 `_llmService.scoreArticle`（底层触发 `scoreAndSummarize`，但丢弃了摘要数据）。
  接着 `summary_service.dart` 内部又会调用 `_llmService.summarize`（底层**再次触发** `scoreAndSummarize`，但丢弃了评分数据）。
* **影响**：每分析一篇文章，LLM 会被相同的内容请求两次，不仅大幅延长简报生成时间，还会**白白浪费双倍的 Token 成本**。
* **解决建议**：
  在生成流程中直接调用 `_llmService.scoreAndSummarize` 获取组合结果，然后统一更新数据库的评分字段和摘要字段，不再拆分为分别调用 `ScoreService` 和 `SummaryService`（或者在 Service 层做一层基于 Article ID 的缓存机制）。

### 🚨 [Bug-02] 简报文章数量分配逻辑有缺陷
* **表现与原因**：
  在 `briefing_service.dart` 的 `_allocateByRatio` 方法中，系统根据国内外比例分配文章名额：
  ```dart
  final domesticSlots = (dailyCount * domesticRatio).round();
  final internationalSlots = dailyCount - domesticSlots;
  selected.addAll(domestic.take(domesticSlots));
  selected.addAll(international.take(internationalSlots));
  ```
* **影响**：如果国内的高分文章不足 `domesticSlots` 篇（比如只要到了 2 篇，但设定需要 10 篇），多出来的空缺名额**没有动态转移**给国际文章。这会导致用户最终看到的简报总数明显少于设定的 `dailyCount`（比如设置了 20 篇，实际可能只出来 12 篇）。
* **解决建议**：在分配名额后进行二次校验，如果某一边不够，将剩余配额给到另一边以补足总量。

### ⚠️ [Issue-03] HTML 清洗代码重复且有小瑕疵
* **表现与原因**：
  `FetchService._stripHtml` 和 `SummaryService._cleanText` 实现了几乎完全相同的 HTML 标签剥离逻辑。此外，基于正则清洗 HTML 容易出现漏网之鱼或破坏原文本结构的情况。
* **解决建议**：将文本清洗提炼为一个公共 Utils 工具类；有条件的情况下，后期可以引入成熟的 HTML Parser。
